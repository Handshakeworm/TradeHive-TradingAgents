# 原开源项目baseline：
## 介绍
所有情况下，自带的记忆功能都是关闭的

| 阶段 | 节点 | 模型 | 读取的数据 | 实际职能 |
|------|------|------|-----------|---------|
| 投资辩论 | Bull / Bear Researcher | quick | 四份报告（技术面、情绪面、新闻、基本面）+ 历史记忆 + 辩论历史。首轮读取报告，后续轮次仅依赖辩论历史和记忆 | 辩论标的是否值得投资 |
| 投资裁决 | Research Manager | deep | 完整辩论历史 + 四份报告（用于检索记忆）+ 历史记忆 | 总结辩论、做出 Buy/Sell/Hold 决策、制定投资计划 |
| 交易转化 | Trader | quick | Research Manager 的投资计划 + 四份报告 + 历史记忆 | 将投资计划转化为明确的交易信号（BUY/HOLD/SELL） |
| 风险辩论 | Aggressive / Conservative / Neutral | quick | Trader 的交易计划 + 辩论历史。首轮读取四份报告，后续轮次仅依赖辩论历史 | 评估 Trader 决策的风险合理性，辩论策略应偏激进还是保守 |
| 最终决策 | Portfolio Manager | deep | 风险辩论完整历史 + Research Manager 的投资计划 + 四份报告（用于检索记忆）+ 历史记忆 | 综合风险辩论，给出五级评级 + 入场策略 + 仓位 + 风险水位 + 时间窗口 |


原项目数据首先不支持回测，很多tools返回实时快照，保留了原先功能的基础上进行了替换数据源

因为只做回测，所以做了本地缓存机制，需要提前拉取5年数据（系统时间往前五年）到本地

传给大模型的数据条数等尽可能与原样保持一致

## 回测逻辑：硬编码映射规则
Rating	操作
BUY	全仓买入（100%）
OVERWEIGHT	买到 60%
HOLD	不动
UNDERWEIGHT	调到 30%
SELL	全部卖出（0%）

从结果上发现决策频率更低效果会更好，这里我们使用的是5天一运行，今天收到分析报告，明天一早下单

但这个持仓信息只在 simulator 内部，Agent 做决策时完全不知道当前持仓是多少（知道了也没用，因为硬编码操作，multi-agent系统只决定当前总仓位应该是多少）

## 修改各节点职能，编写新的回测执行方式：

持仓数据（由回测引擎维护，每日注入 agent state）：

| 字段 | 含义 | 用途 |
|------|------|------|
| current_position_pct | 当前仓位占总资金比例（0-100%） | agents 判断能否加仓/需否减仓 |
| avg_cost | 持仓均价 | agents 评估当前盈亏 |
| total_capital | 账户总资产（现金+持仓市值） | Trader 计算仓位比例的基准 |
| current_stop_loss | 当前止损价 | 前一日 PM 设定，当日检查是否触发 |
| current_take_profit | 当前止盈价 | 前一日 PM 设定，当日检查是否触发 |
| last_action | 上一次操作（Buy/Sell/Hold） | agents 了解最近交易方向 |
| unrealized_pnl_pct | 未实现盈亏百分比 | 由 avg_cost 和当前价算出，直接给 agents |

回测引擎每日流程：
1. 开盘：以开盘价执行前一日 PM 的仓位变化指令
2. 盘中：用当日高低价检查止盈/止损是否触发 → 如触发，以触发价平仓（止盈止损优先级高于 PM 指令，即便刚开盘买入，盘中触发止损也执行）
3. 收盘后：以收盘价更新持仓数据（含 unrealized_pnl_pct），注入 agent state，跑完整 pipeline
4. 保存 PM 输出的新指令，留待次日执行

Buy/Hold/Sell 在不同持仓状态下的含义（满仓 = 可用资金不足以再买入一股）：
- 无仓位：Buy=建仓，Hold=不动，Sell=不动
- 有仓位未满仓：Buy=加仓，Hold=不动，Sell=减仓
- 满仓：Buy=不动，Hold=不动，Sell=减仓

| 阶段 | 节点 | 模型 | 结构化输出 | 输出 | 改动思路 |
|------|------|------|-----------|------|---------|
| 投资辩论 | Bull / Bear Researcher | Flash | 否 | 方向性论据（自由文本） | 不变，继续辩论值不值得投资 |
| 投资裁决 | Research Manager | Deep | **是** | 方向判断 + 投资论点 | 聚焦方向性决策：只输出 Buy/Sell/Hold + 核心理由，不再制定具体计划 |
| 交易规划 | Trader | Deep | **是** | 具体交易方案 | 重新定位为执行规划者：基于 Research Manager 的方向，输出目标仓位%、止盈价、止损价 |
| 风险辩论 | Aggressive / Conservative / Neutral | Flash | 否 | 对交易方案的量化调整建议（自由文本） | 从定性改为定量辩论：审视 Trader 给出的仓位比例、止盈/止损价位是否合理，三方从激进、保守、中性角度辩论参数调整以控制风险 |
| 最终决策 | Portfolio Manager | Deep | **是** | 可直接执行的交易指令 | 输出结构化结果，可直接喂给回测引擎 |

结构化输出 Schema：

**Research Manager:**
```json
{
  "action": "Buy | Sell | Hold",
  "reasoning": "核心理由摘要"
}
```

**Trader:**
```json
{
  "action": "Buy | Sell | Hold",
  "target_position_pct": 40,
  "take_profit_price": 155.00,
  "stop_loss_price": 138.00,
  "reasoning": "交易方案说明"
}
```

**Portfolio Manager（最终输出，回测引擎直接解析）：**
```json
{
  "action": "Buy | Sell | Hold",
  "target_position_pct": 35,
  "take_profit_price": 152.00,
  "stop_loss_price": 140.00,
  "reasoning": "最终决策说明"
}
```

注：`take_profit_price` 和 `stop_loss_price` 在无仓位且不建仓时为 null。


## agent系统层面更改：
### 1. 结构化输出保证

##### 两层保障

**第一层：`with_structured_output()` — 约束 + 校验一体化**
- 传入 Pydantic schema，底层自动用 tool calling / JSON mode 约束模型输出，同时校验字段类型、取值范围、业务语义
- 仅 DeepSeek V3.2 的三个决策节点需要，Qwen 3.5 Flash（辩论节点）输出自由文本无需约束

**第二层：兜底重试**
- 三个结构化节点（Research Manager、Trader、Portfolio Manager）均需
- **校验失败**：将 ValidationError feedback 回模型重新生成
- **调用失败**（网络超时、rate limit、5xx）：指数退避重试，最多 3 次

##### 实施步骤

```
1. 定义 Pydantic schema：
   - Research Manager: action + reasoning
   - Trader: action + target_position_pct + take_profit_price + stop_loss_price + reasoning
   - Portfolio Manager: 同 Trader schema

2. 三个决策节点接入 with_structured_output(schema, method="json_schema")

3. 加兜底重试（校验失败 feedback 重试 + 调用失败指数退避最多 3 次）
```

注意：报告类输出（分析师报告等）不宜过度约束格式，仅决策类输出需硬限制。
