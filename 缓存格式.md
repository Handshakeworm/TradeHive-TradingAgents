记忆系统之后要怎么做，回测要不要考虑记忆：
这是个很好的问题，我的判断是：大概率会下降，或者至多持平。让我解释一下为什么。
核心矛盾：金融市场的非平稳性
你这套记忆机制的底层假设是"相似的情景会产生相似的结果"，但股票市场恰恰是一个regime不断切换的系统。2022年的加息恐慌和2024年的加息恐慌，表面文本高度相似，但市场结构、流动性、持仓分布完全不同。BM25匹配到的"最相似过去"，很可能正好是当下最危险的参照。
几个具体的风险点：
第一，BM25是纯词频匹配，它捕捉的是表面文本相似度，不是市场状态相似度。比如"CPI超预期"这个情景，在不同周期里含义完全相反——通胀上行初期是利空，但市场已经price in之后反而是利好。BM25无法区分这种语义层面的差异。
第二，反思质量本身有问题。LLM生成的"反思"本质上是对结果的事后归因，而金融决策的盈亏里有大量随机成分。一笔赚钱的交易，可能决策逻辑完全错误只是运气好，但系统会把它记录为正面经验并在未来强化类似决策。这就是用噪声训练噪声。
第三，记忆会引入一种"锚定效应"。当系统检索到过去某次相似情景亏了30%，它会变得过度保守；反之则过度激进。这种行为在回测里很难发现，因为回测的路径是确定的，但在实盘中，这种路径依赖会放大尾部风险。
第四，各节点维护各自记忆，意味着系统整体的行为变得极难预测和调试。节点A的记忆可能让它偏保守，节点B偏激进，组合起来的行为不是你设计时能预见的。
如果一定要用记忆，怎么做会更好一些：
考虑把BM25换成对市场状态的结构化编码匹配——不是匹配文本，而是匹配波动率区间、趋势状态、流动性指标这些可量化的特征。反思部分也需要去噪，比如只保留连续多次相同情景下一致方向的结论，过滤掉单次事件的归因。另外，加一个记忆衰减机制，越旧的记忆权重越低，因为市场regime确实会漂移。
不过说到底，最有效的验证方式还是做严格的A/B对比回测

项目 Tools 传给大模型的数据总结
这个项目共有 9 个 tool，分为 4 大类，通过 LangChain 的 bind_tools 传给 LLM。每个 tool 的返回值（字符串）会作为 tool call 的结果回传到 LLM 对话上下文中。

1. Core Stock APIs（核心股价数据）
Tool	使用者	传给LLM的数据	数据量估算
get_stock_data	Market Analyst	指定日期范围的 OHLCV（Open/High/Low/Close/Volume）CSV 格式数据 + 头部信息（ticker、日期范围、总记录数）	约 1-5 KB（30天约30行CSV）
2. Technical Indicators（技术指标）
Tool	使用者	传给LLM的数据	数据量估算
get_indicators	Market Analyst	每个指标逐日的数值列表（如 2024-01-15: 72.3），外加该指标的文字描述。支持的指标有：close_50_sma, close_200_sma, close_10_ema, macd, macds, macdh, rsi, boll, boll_ub, boll_lb, atr, vwma, mfi	每个指标约 1-2 KB（默认30天回看）；LLM 最多调用 8 个指标，合计约 8-16 KB
3. Fundamental Data（基本面数据）
Tool	使用者	传给LLM的数据	数据量估算
get_fundamentals	Fundamentals Analyst	公司概况：名称、行业、市值、PE、EPS、股息率、Beta、52周高低、营收、净利润、利润率、ROE、ROA、负债率等 ~27 个字段	约 0.5-1 KB
get_balance_sheet	Fundamentals Analyst	资产负债表 CSV（季度/年度，通常4列即最近4个季度）	约 3-8 KB
get_cashflow	Fundamentals Analyst	现金流量表 CSV（季度/年度，通常4列）	约 3-8 KB
get_income_statement	Fundamentals Analyst	利润表 CSV（季度/年度，通常4列）	约 3-8 KB
Fundamentals Analyst 一次调用全部 4 个 tool，合计约 10-25 KB。

4. News Data（新闻数据）
Tool	使用者	传给LLM的数据	数据量估算
get_news	News Analyst, Social Media Analyst	特定股票的新闻列表：每条包含标题、摘要、来源、链接（最多20条，按日期过滤）	约 2-10 KB
get_global_news	News Analyst	全球宏观经济新闻：通过4个关键词搜索（stock market、Fed利率、通胀、全球市场），去重后返回标题+摘要+来源+链接	约 2-8 KB（默认5-10条）
get_insider_transactions	（已导入但未被任何 analyst 使用）	内部人交易记录 CSV：包含交易日期、内部人姓名、买卖类型、数量等	约 1-5 KB
各 Agent 使用的 Tools 汇总
Agent	绑定的 Tools	估算总数据量
Market Analyst	get_stock_data + get_indicators（最多8个）	~10-20 KB
Fundamentals Analyst	get_fundamentals + get_balance_sheet + get_cashflow + get_income_statement	~10-25 KB
News Analyst	get_news + get_global_news	~4-18 KB
Social Media Analyst	get_news	~2-10 KB
注意：这些数据都是以纯文本字符串（CSV 或 Markdown 格式）返回给 LLM 的。每次 tool call 的结果都会被追加到 LangGraph 的 messages 状态中，作为对话历史的一部分传给下一次 LLM 调用。所以如果 Market Analyst 调用了 1 次 get_stock_data + 8 次 get_indicators，LLM 的上下文中就会累积约 9 条 tool result message。
